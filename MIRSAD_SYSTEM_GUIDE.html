01 System Foundation
Login & Role-Based Access
Controls how users authenticate to the system and how their role determines what they can see and do across all modules.

Authentication Process

  1. User navigates to the Mirsad web application and is presented with a login screen.

  2. User enters their registered email address and password.

  3. System validates credentials against the database. If correct, a signed JWT token is returned and stored securely in the browser.

    The token expires after the configured session duration. Users must re-login when it expires.

  4. System reads the user's assigned role and project from the token payload.

  5. User is redirected to the Dashboard. Navigation menu items are shown or hidden based on the user's role.

  6. On each API request, the server validates the JWT token and role permissions before processing the request. Unauthorized access returns a 403 error.

  7. User logs out — the token is cleared from the browser and the session ends immediately.

Role Assignment

Only Admins can create and assign roles. Users cannot self-register. The Admin creates user accounts through the User Management panel and assigns a role and project at account creation.

Session Management

Token Behavior
  • JWT stored in browser session
  • Automatically sent with every API call
  • Expires after configured duration
  • Cleared on logout or expiry
Access Guards
  • Route-level protection in the UI
  • API-level authorization on every endpoint
  • Project-scoped data isolation
  • Invalid tokens auto-redirect to login

Status Flow

Anonymous (Not Logged In) → Login → Authenticated (Role Assigned) → Logged Out

04 Daily Operations
Client Request Lifecycle
Manages requests submitted by ministry clients for catering, supplies, events, housekeeping, and other services. Tracks the full lifecycle from submission to delivery confirmation.

Roles Involved

RoleAction
ClientCreates request, confirms delivery
Supervisor / Assistant Supervisor / Project Manager / AdminAssigns, starts fulfillment, marks delivered, can reject

Request Types

Available Request Types
  • Catering (طعام)
  • Maintenance (صيانة)
  • Supplies (مستلزمات)
  • Event (فعالية)
  • Housekeeping (تنظيف)
  • Other (أخرى)
Priority Levels
  • 🔴 Urgent — immediate action required
  • 🟠 High — same-day resolution
  • 🟡 Medium — within schedule
  • ⚪ Low — non-urgent

Step-by-Step Process

  1. Client submits a new request — fills in title, type, description, priority, expected delivery date, and optional item list (name, quantity, unit).

  2. System records the request with status Submitted and logs the creation in the audit trail.

  3. Supervisor / Manager reviews the new request on the Client Requests page or Dashboard. Selects an assignee and clicks Assign.

  4. Status changes to Assigned. The assigned team member is notified.

  5. Assigned user begins work and clicks Start Fulfillment. Status changes to In Progress.

  6. When service/items are delivered, the manager clicks Mark Delivered. Status changes to Delivered.

  7. Client logs in, reviews the delivery, and clicks Confirm Delivery. Status becomes Confirmed. Workflow ends.

  8. Optional — Rejection: At any stage before Confirmed, a manager can reject the request with a written reason. Status changes to Rejected.

Status Flow

Submitted → Assigned → In Progress → Delivered → Confirmed  / Rejected
Note: Client Requests do not automatically deduct inventory. If physical items are consumed (e.g., supplies, catering materials), a separate inventory transaction or operation request must be created to record the consumption.

05 Daily Operations
Coffee Break Request Workflow
Manages daily and event-based coffee break and hospitality service requests across ministry floors. Designed for speed — requests move quickly from submission to floor delivery.

Roles Involved

RoleAction
Supervisor / Assistant Supervisor / Project Manager / AdminCreate request, assign, manage delivery
Supervisor / Project ManagerApprove and mark delivered

Step-by-Step Process

  1. Authorized user creates a Coffee Break Request — specifies floor, requested items (coffee type, quantities, pastries, etc.), expected delivery time, and priority.

  2. System creates the request with status Submitted and shows it on the Dashboard Coffee Break widget.

  3. Supervisor/Manager assigns the request to a service team member. Status → Assigned.

  4. Service team member begins preparation and service. Status → In Progress.

  5. Items are delivered to the floor. Manager marks Delivered. Status → Delivered.

  6. Request is confirmed (by supervisor or auto-confirmed). Status → Confirmed.

Status Flow

Submitted → Assigned → In Progress → Delivered → Confirmed

Inventory Impact

Automatic Deduction: When a Coffee Break Request is marked Delivered, the system automatically deducts the specified quantities from the applicable inventory balances (coffee beans, milk, sugar, pastry items, etc.) for the relevant project period. A stock movement record of type issued is created.

Dashboard Visibility

  • Coffee Break Requests open count shown in the KPI row on the Dashboard.
  • Latest 5 requests shown in the Coffee Break Requests table widget.
  • Export available as Excel from the dashboard widget header.

06 Daily Operations
Operation Request Workflow
Handles internal operational requests for meal preparation, floor distribution, supplies dispatch, and daily cafeteria service execution. Triggered by the operations team — not by clients.

Roles Involved

RoleAction
Supervisor / Assistant Supervisor / AdminCreate and manage operational requests
Project Manager / AdminApprove and confirm completion

Step-by-Step Process

  1. Operations staff creates a new Operation Request — specifies request title, target floor, items needed, quantities, and priority.

  2. System creates the request with status Submitted. Appears on the Dashboard Operation Requests widget.

  3. Supervisor reviews the request and assigns it to available staff. Status → Assigned.

  4. Staff begins the operation (preparation, distribution, etc.). Status → In Progress.

  5. Operation is completed — items delivered to the floor. Manager marks Delivered. Status → Delivered.

  6. Request confirmed by supervisor or project manager. Status → Confirmed.

  7. If an issue arises, request can be Rejected with a rejection reason recorded for audit.

Status Flow

Submitted → Assigned → In Progress → Delivered → Confirmed  / Rejected

Inventory Impact

Automatic Deduction: When an Operation Request is confirmed as Delivered, the specified food or material quantities are automatically deducted from the inventory balance for the project period. The system records a stock movement of type consumed.
Daily Plan Link: Operation Requests can optionally be linked to a Daily Plan. When linked, the planned quantities are pre-filled from the daily plan line items, reducing data entry errors.

07 Procurement & Receiving
Purchase Order Workflow
Manages monthly purchase orders issued to approved suppliers for food and materials. Tracks approved quantities, received quantities, distributed quantities, consumption, and variance — all linked to the monthly period.

Roles Involved

RoleAction
Admin / Supervisor / Project ManagerCreate, edit, and manage purchase orders
Admin / Project ManagerClose purchase orders
All operational rolesView purchase orders and progress

Purchase Order Structure

PO Header
  • PO Number (auto-generated)
  • Supplier (from approved list)
  • Project
  • Month (e.g., 2024-11)
  • Start date / End date
  • Status
  • Notes
PO Line Items
  • Item (food or material)
  • Unit (pcs, kg, L, etc.)
  • Approved Qty
  • Received Qty (auto)
  • Distributed Qty (auto)
  • Consumed Qty (auto)
  • Remaining Qty (auto)
  • Variance (auto)

Step-by-Step Process

  1. Admin/Supervisor creates a new Purchase Order — selects supplier, project, month, and adds line items with approved quantities.

  2. PO is saved with status Active. PO number is auto-generated (e.g., PO-2024-11-001).

  3. Supplier delivers goods → a Receiving record is created referencing this PO. Each received quantity is recorded per line item.

    The system automatically updates ReceivedQty on each PO line when a Receiving record is confirmed.

  4. Status updates automatically as quantities are received: Partially Received when some lines are received, Fully Received when all lines reach 100%.

  5. As items are consumed (through Operation Requests and Coffee Break deliveries), ConsumedQty and RemainingQty are updated automatically on each line.

  6. System monitors remaining quantities. Status changes to Near Depletion when remaining falls below threshold, or Over Consumed if consumption exceeds approved quantity.

  7. At end of period, Admin or Project Manager closes the PO. Status → Closed. Variance is finalized.

Status Flow

Active → Partially Received → Fully Received → Near Depletion  / Over Consumed → Closed
Automatic Inventory Update: Receiving confirmation automatically increases ReceivedQty on PO lines and updates the project's inventory balance (openingBalance + receivedQty). Consumption events reduce RemainingQty and update consumedQty.

08 Procurement & Receiving
Receiving Workflow
Records the physical delivery of goods from suppliers against approved Purchase Orders. Each delivery is inspected, quantities are verified, item conditions are recorded, and inventory is updated automatically upon confirmation.

Roles Involved

RoleAction
Supervisor / Assistant Supervisor / Project Manager / AdminCreate and submit receiving records
Supervisor / Project Manager / AdminConfirm or reject receiving records

Step-by-Step Process

  1. Supplier arrives with goods. The receiving officer opens a new Receiving record and links it to the relevant Purchase Order.

  2. System pre-fills PO line items with ordered quantities. The officer adjusts received quantities per item based on actual delivery.

  3. For each line, the officer records:

    • Quantity received
    • Condition: Good / Damaged / Rejected
    • Batch number (optional)
    • Expiry date (for food items)
    • Notes (for damaged or short deliveries)

  4. Invoice number is entered. The record is saved with status Pending.

  5. Supervisor/Manager reviews the receiving record and clicks Confirm. Status → Confirmed.

  6. System automatically:

    • Updates PO line ReceivedQty for each confirmed line item
    • Updates PO status (Partially/Fully Received)
    • Increases inventory balance (receivedQty) for the project period
    • Creates a Batch record for each food item (FIFO/FEFO tracking)
    • Creates a Stock Movement record (type: received)

  7. If delivery is partly acceptable, status can be set to Partial — only confirmed lines affect inventory.

  8. If delivery is fully rejected, status → Rejected. No inventory impact. A rejection reason is recorded.

Status Flow

Pending → Confirmed  / Partial  / Rejected
Damaged Items: Damaged line items do not increase usable inventory. They are recorded separately as damagedQty and flagged for corrective action or supplier claim.

09 Inventory Management
Food Inventory Workflow
Tracks monthly food inventory balances per project. All received goods, consumption events, spoilage, and transfers are recorded and reflected in real-time inventory balances with FIFO/FEFO rotation enforcement.

Roles Involved

RoleAction
Supervisor / Project Manager / AdminView, manage, and adjust inventory balances
Supervisor / Assistant SupervisorRecord stock movements, transfers, spoilage
AdminSet monthly limits, manage items and categories

Inventory Balance Fields (per Item per Period)

FieldDescriptionUpdated By
Monthly LimitMaximum approved quantity for the monthAdmin (set)
Opening BalanceCarryover from previous periodSystem (auto)
Received QtyTotal goods received from suppliersReceiving confirmation
Consumed QtyQuantity used in service operationsOperation / Coffee Break delivery
Issued QtyQuantity issued via floor transfersTransfer confirmation
Damaged QtySpoiled or damaged items recordedSpoilage / Receiving
Returned QtyItems returned to warehouseManual return entry
Remaining QtyCurrent usable stock = Opening + Received − Consumed − Issued − Damaged + ReturnedAuto-calculated

Inventory Status Logic

StatusCondition
AvailableRemaining Qty > 20% of Monthly Limit
Low StockRemaining Qty between 1–20% of Monthly Limit
Out of StockRemaining Qty = 0
Over ConsumedConsumed Qty > Approved / Monthly Limit

FIFO / FEFO Batch Tracking

  • Each received food item creates a Batch record with batch number, expiry date, and storage zone.
  • When items are consumed or issued, the oldest batch (by expiry date — FEFO) is depleted first.
  • Batches near expiry (within 3 days) trigger Spoilage Alerts visible on the Dashboard.
  • Expired or spoiled batches are marked and deducted from usable inventory.

Status Flow

Available → Low Stock → Out of Stock  / Over Consumed

Transfer Workflow (Warehouse → Floor)

  1. Supervisor/Assistant creates a new Transfer — selects the target floor and adds line items with quantities to be issued from the warehouse.

  2. Transfer is saved with status Draft. Items are not yet deducted.

  3. Supervisor reviews the transfer and clicks Confirm. Status → Confirmed.

  4. System automatically deducts the transferred quantities from the warehouse inventory balance (issuedQty increases, remainingQty decreases). A Stock Movement record of type issued is created.

  5. If transfer is cancelled, status → Cancelled. No inventory impact.

Transfer Status Flow

Draft → Confirmed  / Cancelled

Inventory Status (same as Food)

Available → Low Stock → Out of Stock
Shortage Alerts: When any material item falls below its set minimum level, it appears in the Low Stock Items list on the Dashboard and in the Materials Inventory bar on the Inventory Status widget.

11 Facility & Quality
Maintenance Request Workflow
Handles all facility maintenance and repair requests — from minor equipment faults to critical infrastructure issues. Requests are categorized, prioritized, assigned to technicians, and tracked to resolution.

Roles Involved

RoleAction
Any operational roleReport a maintenance issue (create request)
Supervisor / Project Manager / AdminReview, assign, and close requests
Assigned technicianUpdate progress and mark resolved

Request Categories & Priorities

Categories
  • Electrical
  • Plumbing
  • HVAC (Air Conditioning)
  • Equipment
  • Cleaning
  • Structural
  • Other
Priority Levels
  • 🔴 Critical — immediate shutdown risk
  • 🟠 High — major impact, same-day fix
  • 🟡 Medium — scheduled within 48h
  • ⚪ Low — routine maintenance

Step-by-Step Process

  1. Staff member reports an issue by creating a Maintenance Request — title, description, location (building/floor), category, and priority.

  2. System records the request with status Open and shows it on the Maintenance Requests page and Dashboard counter.

  3. Supervisor/Manager reviews the request and assigns it to a technician. Status → Assigned. Assigned date is recorded.

  4. Technician begins work. Status updated to In Progress.

  5. Technician resolves the issue and records the resolution description. Status → Resolved. Resolution date is recorded.

  6. Supervisor reviews the resolution and closes the request. Status → Closed.

  7. Notes/photos can be added at any step for audit and follow-up purposes.

Status Flow

Open → Assigned → In Progress → Resolved → Closed
Corrective Actions: High or critical maintenance issues that impact food safety can automatically generate a Corrective Action record, which is tracked separately in the Food Safety module until fully resolved.

12 Facility & Quality
Approval Process
The approval process governs Floor Check reports and other regulated operational records, ensuring that data is reviewed, verified, and authorized through a structured chain of command before being finalized.

What Requires Approval

  • Floor Checks — daily inspection reports submitted by supervisors per floor and shift.
  • Receiving Records — delivery confirmations require supervisor sign-off.
  • Transfers — warehouse-to-floor transfers require supervisor confirmation.
  • Client Request Delivery — client must confirm receipt to close the request.

Floor Check Approval Chain

  1. Assistant Supervisor creates a Floor Check for a specific floor and shift. Fills in actual quantities vs. planned quantities for each item. Status → Draft.

  2. Assistant Supervisor submits the report. Status → Submitted. An approval record is created.

  3. Supervisor reviews the submitted floor check. Status → Under Review.

  4. Supervisor can:

    • Approve → Status changes to Approved. Inventory balances are finalized.
    • Return → Status changes to Returned. Notes given to assistant for correction.
    • Reject → Status changes to Rejected. Reason is recorded.

  5. If Returned, the assistant supervisor corrects the data and re-submits. Process repeats from step 2.

  6. Once Approved, the check can be Closed by the Supervisor. Status → Closed. Inventory deductions are committed.

Floor Check Status Flow

Draft → Submitted → Under Review → Approved → Closed
↓ Return ↓ Reject Returned  / Rejected

14 Reporting & Compliance
Audit Log Workflow
The Audit Log records every significant action performed in the system — who did what, when, and on which record. It provides a tamper-proof trail for compliance, accountability, and investigation.

Roles Involved

RoleAccess
AdminFull audit log — all users, all modules, all time
Supervisor / Project ManagerView audit log (read-only), filter by date and action
Assistant Supervisor / ClientNo access to audit log

What is Logged

Action TypeExamples
CreateNew purchase order created, new receiving record created, new client request submitted
UpdateRequest status changed, inventory balance adjusted, user profile updated
Approve / RejectFloor check approved, receiving record confirmed, request rejected
DeleteRecord removed (logged with previous values)
Login / LogoutUser authenticated or session ended
ExportReport generated or data exported

Audit Log Record Fields

FieldDescription
UserFull name, email, and role of the actor
ActionWhat was done (create, update, approve, reject, delete, login, export)
Entity TypeThe module/record type affected (purchase_order, floor_check, client_request, etc.)
Entity IDThe specific record ID that was modified
TimestampExact date and time (server time, UTC)

Using the Audit Log

  1. Navigate to Audit Log in the main navigation (Admin / Supervisor access only).

  2. Apply filters: date range, specific user, action type, or entity type.

  3. Review the paginated log. Each entry shows user name, action taken, module affected, and timestamp.

  4. For investigation: use the Entity ID to navigate directly to the affected record in the relevant module.

Dashboard — Recent Activity Feed

Live Feed: The Dashboard displays the last 8 audit log entries in the Recent Activity section, showing actor initials, name, action, and time elapsed. This gives supervisors instant visibility into what is happening across the system without opening the full audit log.
Immutability: Audit log records cannot be edited or deleted — even by Admins. They serve as the permanent, tamper-proof record of all system activity for compliance and accountability purposes.
